### OAuth / Auth Configuration

- **`pnpm genconfig`** ‚Äî interactive tool to set up OAuth `.env` config (auto-discover from OIDC well-known endpoint or manual)
- OIDC discovery URLs for common providers:
  - Auth0: `https://DOMAIN.auth0.com/.well-known/openid-configuration`
  - Keycloak: `https://HOST/realms/REALM/.well-known/openid-configuration`
  - Azure AD: `https://login.microsoftonline.com/TENANT/.well-known/openid-configuration`
  - Google: `https://accounts.google.com/.well-known/openid-configuration`
- **Auth0 JWE gotcha**: Auth0 may return JWE (encrypted) tokens instead of JWT ‚Äî PostgREST needs JWT. Fix: set signature algorithm to RS256 in Auth0 app settings and ensure API token format is JWT.
- **Environment variables** (all `VITE_` prefixed):
  - `VITE_OAUTH_CLIENT_ID`, `VITE_OAUTH_AUTH_ENDPOINT`, `VITE_OAUTH_TOKEN_ENDPOINT`, `VITE_OAUTH_REDIRECT_URI`, `VITE_OAUTH_SCOPE`
  - `VITE_OAUTH_USERINFO_ENDPOINT`, `VITE_OAUTH_LOGOUT_ENDPOINT`, `VITE_OAUTH_LOGOUT_REDIRECT`
  - `VITE_OAUTH_AUDIENCE` (required for Auth0 API access)
  - `VITE_API_BASE_URL`, `VITE_API_TYPE` (optional, set to `"supabase"` if applicable), `VITE_SUPABASE_APIKEY`
- **User profile access** via `useAuth()`:
  ```tsx
  const { userInfo, userInfoLoading, userInfoError } = useAuth()
  // userInfo?.name, userInfo?.email, etc. (from OIDC userinfo endpoint)
  ```

### Data Fetching & PostgREST API

- All data access via PostgREST ‚Äî no Supabase client
- API base URL is in `VITE_API_BASE_URL` (currently Neon Data API)
- **Always inspect API responses with `curl` before implementing** ‚Äî never assume response structure
- Use generic `useTable` hook ‚Äî do NOT create table-specific hooks unless explicitly asked
- Database schemas are metadata-driven: use `metadata.table.id_column` and `metadata.table.label_column` ‚Äî never assume column names
- Use `ApiErrorDisplay` component for all API error states
- Use `ConfirmDeleteDialog` + `useConfirmDelete` hook for delete operations
- Tokens expire after 1 hour ‚Äî always get a fresh one for testing

**PostgREST filter operators**: `eq`, `neq`, `gt`, `gte`, `lt`, `lte`, `like` (`*` wildcards), `in.(a,b,c)`, `is.null`, `not.is.null`
Combine with `&`: `?select=id,name&status=eq.active&order=created_at.desc&limit=10`

**API testing workflow** (before implementing any new endpoint):
1. `TOKEN=$(curl -s "https://test-oidc-server.ma532.workers.dev/getaccesstoken?user_id=user2&client_id=public-client")`
2. `curl -s -H "Authorization: Bearer $TOKEN" -H "Accept: application/json" "$API_BASE_URL/{table}?limit=1"` ‚Äî inspect field names, types, casing
3. Test filters/ordering/pagination as needed
4. Test error cases (no auth, bad table name)
5. Only then write frontend code matching the actual response shape

**Common error responses** (for debugging):
- Missing auth ‚Üí `"missing authentication credentials: required authorization bearer token in JWT format"`
- Bad token ‚Üí `"signature error"`
- Missing table ‚Üí `code: "42P01"`, `"relation \"public.x\" does not exist"`

### Lessons Learned (Past Failures)

- **Unit tests passing + TypeScript compiling does NOT mean the site works** ‚Äî always verify in the browser
- **Test with real API data, not mocked data** ‚Äî mocks can hide field name mismatches and type issues
- **Verify data types in API responses** ‚Äî booleans may be `true/false` or `0/1`, numbers may be strings; check before assuming
- **Check naming conventions in API responses** ‚Äî verify field names match expected patterns (snake_case, `{table}_id` vs `id`, etc.); don't assume
- **Test the EXACT user flow that was reported broken** ‚Äî don't just check that the code "looks" fixed
- **Watch for silent failures**: dialog closes but data doesn't change, button clicks but no network request fires, console shows errors nobody checked
- **For mutations (delete/create/update)**: confirm the request is actually sent AND the UI reflects the change ‚Äî both must happen
- **Document unexpected API behavior** when found (field naming surprises, unexpected nulls, etc.) ‚Äî add to Discovery Log

- Never modify files in `src/components/ui/` ‚Äî they are CLI-managed and upgradable

- 2026-02-24 ‚Äî **"Bad authorization state" root cause & fix**: `react-oauth2-code-pkce` sets `loginInProgress=true` in localStorage (key `SC_production_loginInProgress`) when `login()` is called. If the user visits the app again before completing sign-in at the OIDC provider (e.g. opens the URL in a new tab, or goes back to it), the library sees `loginInProgress=true` but no `?code=` in the URL, raises "Bad authorization state", and clears storage. **Fix in `src/routes/login.tsx`**: the `useEffect` simply checks the URL ‚Äî if neither `?code=` nor `?error` is present, it calls `login()`. After the library clears the stale state, `loginInProgress` becomes `false`, the effect re-runs, sees no code/error ‚Üí calls `login()` ‚Üí clean PKCE flow starts. No retry counters, no timers needed.
